Improving Internal Design System

Context

When I started working at Evolving Web, they‚Äôd just transferred their design system from Sketch to Figma. Overtime, I noticed their design process could become more efficient if they fully utilized Figma‚Äôs features, so I volunteered to improve it.

Client: Evolving Web
Project type: Internal design system¬†
Team: Sole designer
Timeline: 1 month ~ Q1 2023

The Problem

Designers were wasting time doing repetitive tasks because the Wireframe and Design Template‚Äîfrom which every project originates‚Äîdidn‚Äôt take full advantage of Figma‚Äôs features. Documentation was tedious, application of components and spacing was inconsistent, and the design to developer and QA handoff wasn‚Äôt as clear as it could be.


The Objective

Improve the design system so that menial tasks are automated and/or expeditious, and the organization of the file is easier for developers and QAs to understand.

The Process

Discover & Define

Primary Research + Insights

I took note of inefficiencies in the design process that my colleagues and I talked about in feedback sessions. Additionally, a meeting was held with designers and developers to understand both perspectives.

Labeling and organizing components in Styleguide & Patterns (to handoff to devs and QAs) took too long and was tedious.

Sometimes developers and QAs had trouble finding what they were looking for in Styleguide & Patterns.

Due to budget constraints, certain projects lacked mobile and/or tablet page mockups, forcing developers to make design choices that resulted in inconsistent design quality.

Devs were frustrated with inconsistent spacing and pixel values that didn‚Äôt follow any logic.

Devs were experimenting with a Figma plugin called Tokens Studio that allows syncing of design tokens from Figma to code, speeding up frontend development.

Goals

  • Improve the efficiency of the design process.
  • Improve Wireframe and Design Template so that mobile and tablet mockups can always be made even with a small budget.¬†
  • Eliminate confusion for developers and QAs when handing off designs.

Develop & Deliver

To ensure the new system was simple to grasp and adoptable, I created video tutorials that covered all the updates and uploaded them to the agency‚Äôs intranet for future reference. I included notes from the videos too to make searching for information efficient.

Legible Documentation

Separate text styles were made for documentation so that they won‚Äôt be affected when UI text styles change. Label components speed up the documentation process, and purple note components enable communication between designers, developers, and clients. They‚Äôre used to leave final notes on how an element should function, while Figma comments are used for discussions. Please turn on your volume for all videos to listen to the rationale.

Hidden Grids

Hidden grids on the component page show designers how components should be placed on the grid. It answers questions like ‚ÄúDoes this span the full width of the screen, or does it stop at the margins?‚Äù

Responsive Components

Components were made responsive to adjust to changes in component dimensions and screen sizes, and tablet variants were added so developers don‚Äôt have to make design decisions anymore. Even with a small budget, designers can duplicate desktop prototypes and change the variants to produce mobile and tablet prototypes in minutes.

Style Tokens

A Figma plugin called Tokens Studio allows designers to apply, manage, and sync design tokens from Figma to code exponentially speeding up frontend development.¬†

It‚Äôs useful for designers too because Figma styles are limited to text and colours, but tokens allow global changes to shadows, border radius, border width, spacing, and so much more. Imagine instead of clicking into every variant to adjust a style, designers can simply change the token value, and have it adjust across the entire document, page, or selection.

Spacing Tokens

Developers voiced their frustrations with inconsistent spacing between elements, so I suggested we adhere to the widely used 8-point grid system, and I made spacing tokens to ensure consistency. So imagine instead of going page by page to adjust spacing, designers can simply change the values for each use case in seconds.

Automatic Documentation

I linked text documentation to tokens so they‚Äôd automatically update when the values change. For example, designers no longer need to label hex codes or font styles since design tokens will automatically update them.

Styleguide & Patterns Navigation

Styleguides & Patterns is a prototyped page in Figma that we send to devs and QAs. It covers everything from font styles to grid layouts to components used.

The main navigation was redone and better organized to match the atomic design system that the agency uses, which makes it easier for devs and QAs to find what they‚Äôre looking for.

Styleguide & Patterns Documentation

Before, components were laid out manually. Everytime, more components were added, we had to shift everything else on the page one by one. Now, components are organized with autolayout which means we can shift them using arrow keys, or pop them in between other components while maintaining the spacing. The documentation itself has options to add colour options, text, and icons, which the devs are familiar with.

Outcomes

The UI team found tokens limited their creativity because in the process of experimenting, they broke the link between tokens and main components. There was miscommunication about how tokens should be used in the design process because time was not alloted for designers to watch the tutorials. To resolve the problem, I had to reiterate that experimentation during the art direction phase should be done with instances instead of main components which were already linked to tokens. We also concluded that if a project only requires a few pages, tokens may not be necessary at all since they‚Äôre most useful for large scale projects.¬†

Overall, the system was praised for its use of autolayout, responsiveness, and clarity. Because components were dynamic and made to fit all three devices, and autolayout and label components were utilized, prototyping and documentation were more efficient and less overwhelming. The UX team appreciated the tokens ability to make global changes, and developers found tokens sped up the deployment and updating of simple frontend code.
